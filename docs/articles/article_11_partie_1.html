Cartographie Locale & Algorithme BFS — Partie 1 : La Carte de Similarité Locale

Localiser plusieurs objets à la fois

Série Recherche Visuelle & Localisation d'Objets sans Supervision Dans cette première partie de notre onzième article, nous passons de la théorie du vecteur de concept à sa mise en œuvre pratique sur le terrain. Nous verrons comment comparer notre vecteur de requête $q$ (qui représente l'objet recherché) à chaque patch d'une image de test afin d'obtenir une grille bidimensionnelle de scores de similarité. Cette carte locale révèle où se cachent les structures sémantiques ciblées.

🔍 Étape 1 : Diviser pour analyser

Lorsque nous présentons une image de test dans laquelle nous voulons localiser un objet (par exemple, une photo de rue de $128\times128$ pixels contenant une voiture), l'encodeur de contexte du modèle (le Vision Transformer) commence par la découper en patches.

Si notre modèle utilise des patches de taille $16\times16$ pixels, l'image de test est divisée en une grille de :

$$\frac{128}{16} \times \frac{128}{16} = 8 \times 8 = 64 \text{ patches}$$

Chaque patch $i$ est traité par les couches d'attention du ViT pour générer un embedding local contextualisé $v_i \in \mathbb{R}^D$. Nous obtenons ainsi une grille spatiale de 64 vecteurs représentant les caractéristiques locales de notre image de test.

⚡ Étape 2 : Le Scan Conceptuel (Produit Matriciel)

Nous disposons à présent :

  1. D'un unique vecteur de concept normalisé de requête $q \in \mathbb{R}^D$ (le "concept" de roue que nous avons extrait à l'Article 10).
  2. D'une liste de 64 vecteurs de patches de test $\{v_1, v_2, \dots, v_{64}\}$, eux aussi normalisés.

Pour localiser notre objet dans l'image de test, nous devons effectuer une comparaison un-à-tous. Nous calculons la similarité cosinus (qui se résume à un produit scalaire puisque nos vecteurs sont déjà de norme 1) entre $q$ et chaque patch de test $v_i$ :

$$S_i = q \cdot v_i \quad \text{pour } i \in \{1, \dots, 64\}$$

Chaque score $S_i$ est un nombre réel compris entre $-1$ (concepts opposés) et $1$ (concepts identiques). 

  Image de test (128x128 pixels)               Grille de Similarité (8x8 scores)
  ┌───────────────────────────────┐            ┌───────────────────────────────────┐
  │                               │            │ -0.1  -0.2   0.1   0.0  -0.1  -0.2│
  │                               │            │ -0.2   0.0   0.1  -0.1   0.0  -0.1│
  │   ┌──────┐                    │            │ -0.1   0.2   0.0   0.1   0.1  -0.2│
  │   │      │ <-- Roue           │    ───>    │  0.0   0.8   0.8   0.1  -0.1   0.0│  <-- Scores élevés (Roue)
  │   └──────┘                    │            │  0.1   0.7   0.9   0.2   0.0   0.1│
  │                               │            │ -0.1   0.1   0.1   0.0  -0.2  -0.1│
  └───────────────────────────────┘            └───────────────────────────────────┘

Les zones de l'image de test qui contiennent une structure géométrique similaire à notre requête (comme la roue de la voiture) renverront des scores élevés (proches de $0.7$ ou $0.9$). Les zones d'arrière-plan (comme le ciel ou la route) renverront des scores proches de $0$ ou négatifs.

📈 Étape 3 : Du vecteur linéaire à la grille 2D

Les calculs réseaux s'effectuent sur des tenseurs "aplatis" (linéaires) de forme (B, P, D) où $P$ est le nombre de patches et $D$ la dimension latente.

Pour redonner un sens spatial à nos scores de similarité, nous devons réorganiser notre liste de 64 scores en une grille bidimensionnelle de dimensions $H_{grid} \times W_{grid}$ (ici, $8 \times 8$).

Cette grille 2D est la Carte de Similarité Locale. Elle agit comme une carte de chaleur (heatmap) sémantique : elle nous dit précisément quelles coordonnées spatiales de l'image partagent des concepts avec notre requête.

🛠️ Dans les entrailles du code : Générer la carte locale

Voici la logique implémentée dans notre fichier de recherche (unsupervised_search.py) pour extraire et reconstruire cette carte de similarité :

import torch
import torch.nn.functional as F

def generate_similarity_map(model, test_image, q):
    """
    Calcule la carte de similarité 2D entre la requête q et l'image de test.

    Arguments:
        model: Le modèle JEPA pré-entraîné
        test_image: Tenseur de l'image de recherche de forme (1, 3, H, W)
        q: Vecteur de concept normalisé de forme (1, D)
    """
    model.eval()
    with torch.no_grad():
        # 1. Encoder l'image de test pour obtenir les embeddings locaux
        # De forme (1, N_patches, D)
        test_embeddings = model.context_encoder(test_image)

        # 2. Normaliser les embeddings des patches de test sur la dimension D
        test_embeddings_norm = F.normalize(test_embeddings, p=2, dim=-1)

        # 3. Calculer la similarité cosinus par produit matriciel
        # (1, N_patches, D) x (D, 1) -> (1, N_patches, 1)
        similarity = torch.matmul(test_embeddings_norm, q.transpose(-1, -2))
        similarity = similarity.squeeze(-1) # De forme (1, N_patches)

        # 4. Calculer la taille de la grille (ex: sqrt(64) = 8)
        num_patches = similarity.shape[1]
        grid_size = int(num_patches ** 0.5)

        # 5. Redimensionner en grille 2D spatiale (grid_size, grid_size)
        similarity_map = similarity.view(grid_size, grid_size)

    return similarity_map

🧠 Conclusion & Transition

La carte de similarité locale nous donne une grille continue de scores décrivant le degré de parenté de chaque zone avec notre requête.

Cependant, beaucoup de patches d'arrière-plan ont des scores faiblement positifs (par exemple, $0.12$ ou $0.18$) simplement en raison de corrélations géométriques aléatoires ou de textures communes. Si nous gardons tout, notre détection sera parasitée.
