Passage à l'Échelle — Partie 2 : L'Astuce de Vectorisation Batch

Prédire la cohérence spatiale globale sur GPU

Série I-JEPA — Article 3 : Passer à l'échelle (Niveau 2) Dans cette deuxième partie, nous abordons un défi d'optimisation majeur. Lorsque l'on passe d'une seule cible à $M$ cibles disjointes, le piège classique est d'écrire une boucle séquentielle Python sur le GPU. Découvrez comment notre implémentation résout ce problème par une vectorisation élégante et ultra-rapide.

🐢 Le goulet d'étranglement : Le coût des boucles séquentielles

Imaginons que nous ayons un lot (batch) de $B = 128$ images, et que pour chaque image, nous voulions prédire $M = 4$ cibles distinctes.

Si nous écrivions le code naïvement avec une boucle séquentielle en Python :

# CODE NAÏF ET LENT
predictions = []
for m in range(self.num_targets):
    # Récupérer les masques de la cible m
    # Faire une passe complète dans le Predictor
    pred = self.predictor(s_x, target_masks[m])
    predictions.append(pred)

Sur le papier, c'est logique. Mais sur le GPU, c'est une catastrophe :

  1. CPU Overhead : Python va devoir lancer 4 instructions consécutives au GPU. Le temps que le CPU prépare et envoie chaque commande, le GPU (qui calcule en nanosecondes) se retrouve à attendre sans rien faire.
  2. Sous-utilisation du parallélisme : Au lieu de traiter un grand bloc de données en parallèle, le GPU traite 4 petits blocs successivement, ce qui détruit son efficacité.

Pour maximiser le débit (throughput) du GPU, nous devons regrouper toutes ces opérations en une seule passe matricielle parallèle.

🔀 La solution : L'expansion de la dimension de Batch

Pour paralléliser le calcul, l'astuce de notre classe MultiTargetJEPA (models_multi.py) consiste à faire croire au Predictor qu'il traite un batch d'images beaucoup plus grand.

Au lieu de lui passer un batch de taille $B$ (128) et de boucler 4 fois, nous fusionnons la dimension de batch et la dimension des cibles pour lui donner un unique batch virtuel de taille $B \times M$ (soit $128 \times 4 = 512$ éléments) à traiter en une seule fois.

Voici le schéma exact des transformations de formes (shapes) de nos tenseurs dans le code : 

                  s_x (Représentation Contexte) : (B, K_x, D)
                                       │
                                       ▼ (Ajout axe cible)
                                 (B, 1, K_x, D)
                                       │
                                       ▼ (Répétition pour M cibles)
                                 (B, M, K_x, D)
                                       │
                                       ▼ (Aplatissement batch virtuel)
                             s_x_exp : (B * M, K_x, D)
                                       │
                                       ├─► Concaténation ──► pred_inputs : (B * M, K_x + K_y, D)
                                       │
                             s_y_tqk : (B * M, K_y, D) 
                                       │  (Mask Tokens + Positions Cibles)
                                       ▼
                                 [ Predictor ]
                                       │
                                       ▼ (Sortie)
                            pred_outputs : (B * M, K_x + K_y, D)
                                       │
                                       ▼ (Sélection cibles)
                           s_y_hat_flat : (B * M, K_y, D)
                                       │
                                       ▼ (Restauration dimensions)
                             s_y_hat : (B, M, K_y, D)

🔍 Analyse ligne par ligne de la vectorisation (models_multi.py)

Regardons comment cette chorégraphie de tenseurs est écrite en PyTorch dans la méthode forward :

Étape 1 : Dupliquer le contexte sur la dimension de Batch

Le contexte visible $s_x$ (de taille (B, K_x, D)) est identique pour les $M$ cibles d'une même image. Nous devons donc le copier $M$ fois :

# 1. s_x: (B, K_x, D) -> unsqueeze -> (B, 1, K_x, D)
# 2. repeat -> (B, M, K_x, D)
# 3. view -> (B * M, K_x, D)
s_x_exp = s_x.unsqueeze(1).repeat(1, M, 1, 1).view(B * M, K_x, D)

Étape 2 : Préparer les requêtes de position pour toutes les cibles

Nous récupérons les coordonnées GPS spatiales de tous les patches cibles, puis nous fusionnons également ces dimensions pour obtenir la forme (B * M, K_y, D) :

# Extraction des coordonnées GPS des cibles
pos_embed = self.context_encoder.pos_embed # (1, N, D)
idx_pos = flat_target_indices.unsqueeze(-1).expand(-1, -1, D)
target_pos_embed_all = torch.gather(pos_embed.expand(B, -1, -1), 1, idx_pos) # (B, M * K_y, D)

# Passage au batch virtuel
target_pos_embed = target_pos_embed_all.view(B * M, K_y, D)
target_mask_tokens = self.mask_token.expand(B * M, K_y, -1) + target_pos_embed # (B * M, K_y, D)

Étape 3 : La prédiction parallèle et la reconstruction

Nous concaténons le contexte dupliqué et les requêtes de cibles. Le Predictor effectue son attention en une seule passe globale. Ensuite, nous restaurons la géométrie d'origine :

# Concaténation et calcul unique sur GPU
pred_inputs = torch.cat([s_x_exp, target_mask_tokens], dim=1) # (B * M, K_x + K_y, D)
pred_outputs = self.predictor(pred_inputs) # (B * M, K_x + K_y, D)

# On extrait uniquement la partie prédite (les K_y derniers tokens)
s_y_hat_flat = pred_outputs[:, K_x:] # (B * M, K_y, D)

# Retour aux dimensions d'origine pour le calcul de la loss
s_y_hat = s_y_hat_flat.view(B, M, K_y, D) # (B, M, K_y, D)

📊 Le gain de performance mesuré

Grâce à cette vectorisation, le coût de pré-entraînement d'un modèle multi-cibles de I-JEPA est presque identique à celui du modèle mono-cible !

Le GPU calcule l'attention sur le batch virtuel de taille $B \times M$ de manière optimisée en exploitant tous ses cœurs CUDA en parallèle. Il n'y a plus aucune latence d'attente CPU-GPU.

🧠 Conclusion & Transition

La vectorisation nous permet de passer à l'échelle sans sacrifier la vitesse de calcul. Nous disposons maintenant d'un pipeline multi-cibles puissant et optimisé.

Mais pour que notre JEPA apprenne des représentations sémantiques de haute voltige, quelle taille de réseau devons-nous choisir ? Comment le fait d'augmenter la taille de notre Vision Transformer modifie-t-il la qualité des caractéristiques apprises ?

C'est ce que nous allons explorer dans la troisième partie en analysant les options de capacité des modèles (Tiny, Small, Base) et l'impact du scaling !

🛠️ Défi curieux n°2

Imaginez un grand restaurant où 128 tables (la taille de Batch $B$) commandent chacune 4 entrées différentes (les $M$ cibles). Si le serveur fait un aller-retour en cuisine pour chaque entrée l'une après l'autre (boucle séquentielle), les cuisiniers passent leur temps à l'attendre. Si, au contraire, il apporte les 512 commandes d'un coup, la cuisine (les cœurs CUDA du GPU) peut tout préparer en parallèle. À votre avis, si le restaurant a une taille de cuisine limitée (la mémoire VRAM du GPU), quel est le risque de regrouper un trop grand nombre de commandes (par exemple, un batch virtuel géant) par rapport à un service table par table ?
