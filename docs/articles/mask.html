La Stratégie de Masquage

Pourquoi masquer par grands blocs

Ce fichier gère la séparation spatiale de l'image en deux zones distinctes :

  • le Context (ce que le modèle voit)
  • le Target (ce que le modèle doit deviner).
sample_masks(batch_size, grid_size=8, target_size=4, ...)

Ce qu'elle fait :

  • Pour chaque image du batch, elle choisit de manière aléatoire les coordonnées d'un grand bloc carré de taille target_size x target_size (ici 4x4 patches).
  • Elle calcule les index de tous les patches appartenant à ce bloc cible (target_patches).
  • Elle extrait par complémentarité tous les autres patches restants de la grille pour former le contexte (context_patches).
  • Elle retourne ces index sous forme de tenseurs.

Pourquoi cette approche ?

  • Dans I-JEPA, on ne masque pas des patches individuels de manière aléatoire (comme dans MAE - Masked Autoencoders), mais on sélectionne de grands blocs contigus.
  • Si l'on masquait des patches uniques isolés, le modèle pourrait simplement interpoler les pixels voisins (par exemple, compléter une ligne ou une couleur continue).
  • En masquant un grand bloc, on force le modèle à comprendre la structure sémantique globale (par exemple, "si je vois des pattes de chat à gauche et une queue à droite, il doit y avoir un corps au milieu") plutôt que de faire de l'interpolation locale à basse fréquence.

C'est quoi un patch ?

Un patch (ou "bloc d'image") est une petite sous-image carrée obtenue en découpant l'image d'entrée en une grille de morceaux réguliers et non chevauchants.

C'est l'équivalent d'un mot dans une phrase pour les modèles de traitement d'image modernes appelés Vision Transformers (ViT).

Pourquoi fait-on cela ? (Le problème des pixels) Les réseaux de neurones de type Transformer ont été conçus à l'origine pour le texte. Si l'on voulait appliquer un Transformer directement sur les pixels d'une image, ce serait impossible en pratique :

  • Une image standard de 224 x 224 pixels contient 50,176 pixels.
  • L'attention d'un Transformer calcule les relations de chaque élément avec tous les autres. Le coût grandit de manière quadratique (50,176^2 approx 2,5 milliards d'opérations juste pour une couche !).
  • En découpant l'image en patches, on réduit considérablement le nombre d'éléments à traiter.

Illustration concrète avec notre projet CIFAR-10

Dans notre code, nous utilisons les dimensions suivantes :

  • Taille de l'image : 32 x 32 pixels.
  • Taille d'un patch (patch_size) : 4 x 4 pixels.
  • Si on découpe l'image de 32 x 32 pixels en petits morceaux de 4 x 4 pixels, on obtient une grille de 8 patches de large sur 8 patches de haut (32 / 4 = 8).

Chaque patch (de la case 0 à 63) est ensuite aplati et converti en un unique vecteur numérique (un embedding de taille 192 dans notre modèle).

Pour résumer le flux de JEPA avec cette notion :

  • L'image est transformée en 64 patches.
  • La fonction de masquage décide par exemple de masquer un bloc cible de 4 x 4 patches (soit 16 patches, disons les cases 18, 19, 20, 21, 26, 27, 28, 29...).
  • Le Context Encoder ne reçoit que les 48 patches restants (le contexte) et les analyse.
  • Le Predictor doit deviner la signature visuelle des 16 patches masqués à partir des 48 patches visibles.

Les blocs peuvent se chevaucher ou pas?

Il faut distinguer deux niveaux de "blocs" dans I-JEPA : les patches individuels (les petits morceaux de 4x4 pixels) et les blocs cibles (targets) (les groupes de patches que l'on masque).

Voici comment se gère le chevauchement pour chacun d'eux :

  1. Les Patches Individuels (au niveau des pixels) : Non, ils ne se chevauchent pas.

Dans l'encodeur (le Vision Transformer), les patches individuels sont découpés de manière stricte et adjacente sans aucun chevauchement.

Dans notre code models.py (ligne 14), la projection est faite via une convolution 2D : nn.Conv2d(..., kernel_size=patch_size, stride=patch_size)

Comme la taille du noyau (kernel_size) est de 4 et le pas (stride) est de 4, la convolution glisse exactement de 4 pixels en 4 pixels. Les patches sont donc de parfaits petits carrés collés les uns aux autres.

  1. Les Blocs Cibles / Targets (les groupes de patches masqués)

Dans notre prototype et dans le papier officiel I-JEPA, la règle du chevauchement est la suivante :

A. Entre les différentes Cibles (Targets) : Oui, elles peuvent se chevaucher.

Dans notre implémentation simplifiée : Pour rester simple et pédagogique, nous ne choisissons qu'un seul bloc cible par image. Il n'y a donc pas de sujet de chevauchement.

Dans le papier officiel I-JEPA (Meta) : Ils échantillonnent de 4 à 6 blocs cibles par image (de tailles et ratios d'aspect aléatoires). Ces blocs cibles peuvent se chevaucher entre eux. Le modèle doit ainsi prédire plusieurs régions qui peuvent partager des patches communs.

B. Entre le Contexte et les Cibles (Context vs Target) : Non, ils ne se chevauchent jamais.

C'est une règle absolue du framework JEPA.

Le contexte est défini comme l'image entière privée de l'union de tous les blocs cibles.

Si le contexte chevauchait la cible, le modèle tricherait : pour prédire la cible, il lui suffirait de copier les patches du contexte qui se trouvent au même endroit. La tâche de prédiction sémantique serait détruite.
